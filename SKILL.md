# PowerPoint レビュースキル

顧客向け資料として提出・共有する前に、日本語を中心としたPowerPointファイルを多角的にレビューし、改善ポイントをMarkdownレポートとして出力する。

## 対象

- 主に日本語で書かれたPowerPoint資料（.pptx）
- システムエンジニアが作成した顧客向け資料を想定（英単語・英語の関数名・技術用語が混在する場合も許容）
- 対象読者：顧客（IT専門家でない可能性がある）

---

## Step 1: ファイルとレビュー対象ページの確認

ユーザーからPowerPointファイルのパスを受け取る。パスが指定されていない場合は確認する。

**ページ指定について：**
- ページ（スライド番号）が指定されている場合は、そのページのみをレビュー対象とする
- ページ指定はカンマ区切りで複数指定可能（例：`1,3,5,7`）
- 指定なしの場合は全スライドをレビューする

ページ指定の受け取り方の例：
- `proposal.pptx の 2,4,7 ページだけレビューして`
- `--pages 1,3,5 で review して`
- 指定なし → 全スライドが対象

ページ指定がある場合は、Step 3 のテキスト抽出時に `--pages` オプションで渡す。

---

## Step 2: Python環境の確認とpython-pptxのインストール

テキスト抽出には `python-pptx` ライブラリを使う。以下の順で環境確認とセットアップを行う。

### 2-1. Pythonの確認

```bash
python --version
```
または
```bash
python3 --version
```

Pythonが見つからない場合は、ユーザーに以下を案内してこのスキルを中断する：
> Python が見つかりませんでした。[python.org](https://www.python.org/downloads/) からPythonをインストールしてから再実行してください。

### 2-2. python-pptxの確認

```bash
python -c "import pptx; print('ok')"
```

エラーが出た場合（未インストール）は Step 2-3 へ。

### 2-3. プロキシ設定の確認（インストール前に必ず聞く）

インストールが必要な場合、**先にプロキシが必要かどうかをユーザーに確認する**：

> `python-pptx` のインストールが必要です。ネットワーク環境にプロキシ設定は必要ですか？
> - 不要な場合：そのままインストールします
> - 必要な場合：プロキシURLを教えてください（例：`http://proxy.example.com:8080`）

プロキシなしの場合：
```bash
pip install python-pptx
```

プロキシありの場合（ユーザーからURL取得後）：
```bash
pip install python-pptx --proxy http://proxy.example.com:8080
```

インストール後、再度 `import pptx` で確認する。

## Step 3: 用語統一リストの読み込み

`references/terminology.json` に正式表記と誤表記（ゆれ）のリストが定義されている。
このファイルを読み込み、どの単語を「正」とするかを把握しておく。

ファイルの構造：
```json
{
  "terms": [
    {
      "correct": "サーバ",       // 正式表記
      "variants": ["サーバー"],  // 誤表記・ゆれのリスト
      "category": "IT用語",
      "notes": "末尾長音符を省略する表記を正とする"
    }
  ]
}
```

用語リストが存在しない場合は、その旨をユーザーに伝えてレビューを続行する（用語チェックの観点は「一般的な表記ゆれの検出」のみとなる）。

---

## Step 4: テキスト抽出

このスキルに同梱されている `scripts/extract_pptx.py` を使ってスライドのテキストと構造情報を抽出する。

全スライドを対象にする場合：
```bash
python <skill-dir>/scripts/extract_pptx.py <path-to-pptx>
```

特定ページのみ対象にする場合（カンマ区切りでページ番号を指定）：
```bash
python <skill-dir>/scripts/extract_pptx.py <path-to-pptx> --pages 1,3,5
```

出力されるJSONには以下が含まれる：
- `total_slides`: ファイル全体のスライド枚数
- `reviewed_slides`: 実際にレビュー対象としたスライド番号のリスト
- `slides[].slide_number`: スライド番号
- `slides[].title`: スライドタイトル
- `slides[].total_chars`: そのスライドの総文字数
- `slides[].shapes[].shape_kind`: 図形の種別（下表参照）
- `slides[].shapes[].paragraphs`: 段落テキストと見出しレベル（level）
- `slides[].notes`: ノートペイン（スピーカーノート）のテキスト。ノートがない場合は `null`

| shape_kind | 内容 |
|-----------|------|
| タイトル | タイトルプレースホルダー |
| コンテンツ | コンテンツプレースホルダー |
| テキストボックス | 独立したテキストボックス |
| 図形 | AutoShape（四角、矢印など）内テキスト |
| 表 | テーブルセル |
| グラフ | チャートのタイトル・軸ラベル（`グラフタイトル:` / `カテゴリ軸ラベル:` / `値軸ラベル:` プレフィクスつき） |
| SmartArt | SmartArt ダイアグラムのノードテキスト |
| 代替テキスト | 図形の Alt テキスト（アクセシビリティ用テキスト） |
| その他 | 上記以外 |

---

## Step 4: 用語チェックスクリプトの実行

`scripts/check_terminology.py` を使って、抽出したテキストと用語リストを照合する。

```bash
python <skill-dir>/scripts/check_terminology.py <extracted-json> <skill-dir>/references/terminology.json
```

出力JSONには、スライドごとに「どの用語が、どのスライドの、どのコンテキストで誤表記されていたか」が含まれる。
この結果をStep 5の観点1（文章校正・表記ゆれ）に組み込む。

---

## Step 5: レビュー分析

抽出したテキスト全体とStep 4の用語チェック結果を組み合わせて、以下の4つの観点でレビューを行う。

**ノートペインについて：**
`slides[].notes` にテキストがある場合、そのスライドのノートペインにも校正対象のテキストが含まれる。ノートペインのテキストも観点1（表記ゆれ・文法）のチェック対象とする。箇所の示し方は「ノートペイン」と明記する。

### 観点1：文章校正・表記ゆれ

レビューの目的は、読み手（顧客）が混乱せず資料の品質への信頼を損なわないようにすること。

- **用語統一リストによる検出**：Step 4のcheck_terminology.py結果を使い、正式表記でない用語を漏れなく報告する。「どのスライドの、どの箇所で、何が正しいか」を明示する
- **誤字・脱字の検出**：明らかな入力ミスや不自然な文字の欠落
- **リスト外の表記ゆれ**：用語リストに載っていないが、スライド内で同じ概念に複数の表記が混在している場合も指摘する
- **文法エラー**：助詞の誤り、読点の欠落、文として成立していない箇所
- 英単語・英語の関数名・技術用語は**指摘対象外**（許容表記として扱う）

### 観点2：論理的整合性

プレゼン全体の流れとして、聴衆（顧客）が「話が繋がっている」と感じられるかを確認する。

- **スライド間の矛盾**：前のスライドで述べた事実と後のスライドの内容が食い違っていないか
- **主語・述語のねじれ**：文の主語と述語が意味的に対応しているか（例：「課題は～を実施します」のような不自然な接続）
- **因果関係の破綻**：「なぜなら」「そのため」などの接続詞で繋がれた文が、実際に因果関係を成していないケース

### 観点3：構成と可読性

1スライドに情報を詰め込みすぎると顧客が理解しにくくなる。構成の問題を指摘する。

- **情報量の偏り**：`total_chars` を参照し、特定スライドの文字数が突出して多い・少ない場合に指摘する。目安として1スライド300文字超はやや多め、600文字超は要注意とする（タイトルスライドなど例外は除く）
- **見出し階層の適切さ**：段落の `level` 情報から、見出しの深さが適切か、階層が飛んでいないかを確認する
- **1スライド1メッセージ原則**：複数のテーマが混在しているスライドがあればスライド分割を提案する

### 観点4：顧客向け表現の調整

顧客（ITの専門家でない可能性がある）が理解できる表現かどうかを確認する。

- **過度な専門用語の指摘**：技術的な略語（例：「APIM」「RBAC」「IaC」など）を説明なしに使っている場合に指摘する。英語の一般的な技術用語（Server、Cloudなど）は許容
- **トーン＆マナーの統一**：「です・ます調」と「だ・である調」の混在、丁寧さのレベルがスライドによってバラバラな場合に指摘する
- **曖昧な表現**：「など」「等」「場合によっては」が多用されており、顧客に具体性が伝わらない可能性がある箇所

## Step 6: レポートの出力

レビュー結果をMarkdownファイルとして保存する。

### 出力ファイル名と保存先

- **保存先**: PowerPointファイルと**同じフォルダ**
- **ファイル名**: `YYYYMMDD_<元ファイル名（拡張子なし）>.md`
  - 例：`proposal_v2.pptx` → `20260318_proposal_v2.md`
- **重複時**: 末尾に2桁の連番を付与（`_01`, `_02`, ...）
  - 例：`20260318_proposal_v2.md` が既存の場合 → `20260318_proposal_v2_01.md`

ファイル名を組み立てる手順（Pythonで実行することを推奨）：

```python
import os
from datetime import date

pptx_path = "<PowerPointのパス>"
folder = os.path.dirname(os.path.abspath(pptx_path))
stem = os.path.splitext(os.path.basename(pptx_path))[0]
today = date.today().strftime("%Y%m%d")
base_name = f"{today}_{stem}.md"
out_path = os.path.join(folder, base_name)

# 重複チェック
if os.path.exists(out_path):
    i = 1
    while True:
        candidate = os.path.join(folder, f"{today}_{stem}_{i:02d}.md")
        if not os.path.exists(candidate):
            out_path = candidate
            break
        i += 1
```

### レポートの構成

以下のフォーマットで出力する：

```markdown
# PowerPoint レビューレポート

**ファイル**: <ファイル名>
**レビュー日時**: <YYYY-MM-DD>
**総スライド数**: <N> 枚（ファイル全体）
**レビュー対象ページ**: 全スライド  ← ページ指定がある場合は「1, 3, 5 ページ」のように記載
**対象読者**: 顧客向け

---

## 総合サマリー

<全体を通じた主な課題と、優先度が高い改善ポイントを3〜5行で要約する>

---

## スライド別指摘事項

### スライド <N>：<タイトル>

#### 📝 文章校正・表記ゆれ
| 指摘内容 | 箇所 | 改善案 |
|---------|------|--------|
| 表記ゆれ「サーバー」→「サーバ」（用語リストに基づく） | コンテンツ（箇条書き2行目） | 「サーバ」に統一 |

#### 🔗 論理的整合性
- **[ねじれ]** 主語「課題は」に対して述語「改善します」が不自然。体言止め「課題：パフォーマンス低下」に変更を推奨（箇所：コンテンツ 1行目）

#### 📊 構成・可読性
- **[情報量]** 文字数約750文字。要点を絞り、2スライドに分割することを推奨

#### 👥 顧客向け表現
- **[専門用語]** 「RBAC」→ 初出時に「ロールベースアクセス制御（RBAC）」と説明を追加することを推奨（箇所：コンテンツ 3行目）

---

（指摘がないスライドは省略してよい）

---

## 全体的な改善提案

1. <改善提案1>
2. <改善提案2>
3. <改善提案3>
```

### 出力ルール

- **テキストの引用・転載は禁止**: レポートには指摘内容と箇所の説明のみ記載する。スライドのテキスト内容をそのまま貼り付けない
- **箇所の示し方**: `shape_kind`（タイトル／コンテンツ／テキストボックス／図形／表）と位置（「〇行目」「〇列目」など）で示す
- 指摘がないスライドはスライド別セクションに含めない（レポートを見やすく保つため）
- 各観点で問題がなければそのセクションも省略する
- 表形式は読みやすさのために使うが、指摘が1件のみの場合は箇条書きでもよい
- 改善案は「禁止」ではなく「推奨」のトーンで書く（顧客向け資料のレビューは最終的に著者が判断するため）
